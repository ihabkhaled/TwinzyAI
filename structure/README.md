# structure/ — Repository Structure Knowledge Area

This folder is the canonical **structural** description of the TwinzyAI monorepo: what exists,
where it lives, who owns it, how the layers connect, and how the main flows traverse them.
It is part of the canonical plane of the Knowledge OS (see [knowledge/README.md](../knowledge/README.md)).

It describes **structure, not policy**. Engineering rules live in [rules/](../rules/README.md),
the architecture canon in [context/architecture-map.md](../context/architecture-map.md), and
lifecycle governance in [CLAUDE.md](../CLAUDE.md). When this folder and those disagree, they win.

## What lives here

| File | Owns |
| --- | --- |
| [repository-map.md](repository-map.md) | The monorepo at a glance: workspaces and top-level folders |
| [module-catalog.yaml](module-catalog.yaml) | One structured entry per module (responsibility, surface, deps, tests, risks) |
| [ownership-map.yaml](ownership-map.yaml) | Module id → responsibility, owner, delivery lane, canonical docs (compiler input) |
| [layer-map.md](layer-map.md) | Backend and frontend layer models + the ESLint configs that enforce them |
| [runtime-topology.md](runtime-topology.md) | Processes, ports, docker-compose services, SSE flow, the no-DB statement |
| [entrypoint-catalog.yaml](entrypoint-catalog.yaml) | Process/app entrypoints and the bootstrap chain |
| [command-catalog.yaml](command-catalog.yaml) | The important root npm scripts and what they do |
| [configuration-map.md](configuration-map.md) | Where configuration lives on each side |
| [flows/](flows/analyze-flow.md) | End-to-end flow walkthroughs: analyze, share, payment |
| [modules/](modules/api-game.md) | One focused page per module (8 API modules, web game, web ui-preferences, shared) |

## Read order

1. [repository-map.md](repository-map.md) — orient in the tree.
2. [layer-map.md](layer-map.md) — learn the layer models before touching code.
3. [module-catalog.yaml](module-catalog.yaml) / [modules/](modules/api-game.md) — the module you are changing.
4. [flows/](flows/analyze-flow.md) — if your change crosses modules.
5. [runtime-topology.md](runtime-topology.md), [configuration-map.md](configuration-map.md),
   [command-catalog.yaml](command-catalog.yaml) — when running, configuring, or releasing.

## Generated maps live elsewhere

`source-to-test-map.yaml` and `change-impact-map.yaml` are **not** authored here — they are
generated by the knowledge compiler into `.ai/graphs/` from the real source tree. Never hand-edit
them; fix the source or the compiler and rebuild (`npm run knowledge:build`).
