# ADR FE-0002: Component workbench route instead of Storybook (apps/web)

- **Status:** Accepted
- **Date:** 2026-07-06
- **Deciders:** Twinzy frontend maintainers
- **Related:** [ADR FE-0001](adr-fe-0001-strict-next-architecture.md),
  [memory/ui-design-system-decisions.md](../../memory/ui-design-system-decisions.md),
  [rules/02-frontend-components-tsx.md](../../rules/02-frontend-components-tsx.md),
  [docs/mobile-pwa-standards.md](../../docs/mobile-pwa-standards.md)

## Context

The design-system primitives in `apps/web/src/packages/ui-primitives` (Button, Input, Label,
Card, Alert, Spinner, Skeleton, Stack, PageContainer) need a living showcase: a place where
engineers, designers, and reviewers can see every variant and state rendered by the real app.

The default answer is Storybook. But Storybook is a second application inside the repo: it brings
its own bundler, its own React renderer configuration, dozens of transitive dependencies that
Trivy and `npm audit` must clear under our zero-unhandled-vulnerability policy, and a major
version churn cadence that historically breaks on every Next.js or React upgrade. Worst of all,
Storybook stories compile through a _different_ toolchain than production, so a story can render
while the same component breaks under the app's build, CSP, i18n provider, or theme attributes —
the showcase can silently lie.

## Decision

We do not use Storybook. The living showcase is a normal App Router route at `/workbench`
(`apps/web/src/app/(workbench)/workbench/page.tsx`, path constant `ROUTE_PATHS.workbench` in
`apps/web/src/shared/constants/route-paths.constants.ts`).

The workbench page renders every primitive variant — button variants including `secondary`,
`danger`, `ghost`, and `disabled`; Alert tones `info`/`success`/`warning`/`danger`; Spinner and
Skeleton; labeled Input — using the same `@/packages/ui-primitives` imports, the same
`getServerTranslations` copy, and the same class-variant files as product code. It is compiled by
the same `next build`, served behind the same CSP from `apps/web/src/proxy.ts`, themed by the same
`[data-theme='dark']` tokens in `apps/web/src/app/styles.css`, and covered by the same Playwright
accessibility and visual suites (`npm run test:a11y`, `npm run test:visual`).

Adding a primitive to `apps/web/src/packages/ui-primitives` MUST include adding its states to the
workbench page in the same pull request.

### Enforcement

The workbench is a first-class route, so everything that gates product code gates it: the
architecture ESLint plugin, typecheck via `npm run typecheck`, and the e2e/a11y/visual suites
under `apps/web/src/tests/`. Zero additional dependencies, zero extra build pipeline.

## Consequences

### Positive

- The showcase is always true: if it renders in `/workbench`, it renders in production, under the
  production CSP, theme, and locale/RTL handling — which matters because Twinzy is a mobile-first
  PWA shipped in English and Arabic (RTL).
- Zero added dependencies — nothing new for `npm run security:audit` / `npm run security:scan` to
  clear, nothing to migrate on the next React or Next.js major.
- Visual regression baselines for primitives come free from the existing Playwright visual suite
  pointed at one route.

### Negative / accepted costs

- No Storybook addon ecosystem: no controls/knobs panel, no autogenerated props docs, no
  isolated-iframe rendering. Prop documentation lives in TypeScript types and
  [memory/ui-design-system-decisions.md](../../memory/ui-design-system-decisions.md) instead.
- The workbench page grows with the design system and must be curated by hand.
- Non-engineers must run the app (`npm run dev`) or use a deployed environment to browse it; there
  is no separately publishable static showcase.

### Revisit trigger

Re-evaluate with a superseding ADR if either becomes true: (a) the primitive catalog grows past
the point where one hand-curated route can honestly represent all states, or (b) a dedicated
design team needs interactive prop exploration or a publishable component catalog that the
workbench cannot provide.

## Alternatives considered

### Storybook

Industry standard, rich addon ecosystem, designer-friendly. Rejected for dependency weight under
the zero-unhandled-vuln policy, chronic version churn against Next/React majors, and the
parallel-toolchain problem: a showcase compiled differently from production can lie.

### Ladle / Histoire (lighter story runners)

Lighter than Storybook but share the same structural flaw — a second toolchain and story format
outside the App Router, CSP, and i18n providers. Rejected for the same reason at smaller scale.

### Do nothing (no showcase)

Primitives would only be visible scattered across product screens; reviewers could not audit a
variant matrix, and visual baselines would have no stable canvas. Rejected.
