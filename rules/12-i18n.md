# 12 — i18n & Messaging

> Two surfaces, one discipline: the **frontend** renders every user-facing string from the typed dictionary; the **backend** never returns human sentences as a contract — it returns stable `messageKey`s the frontend maps to localized copy. Implements rules 25 and 29 of [00-non-negotiable-rules.md](./00-non-negotiable-rules.md).

---

## Part A — Frontend strings (the typed dictionary)

- All user-facing strings live in `apps/web/src/i18n/en.ts` and render via `t(key)`.
- Keys are compile-time checked (`TranslationKey`) — duplicates and typos are TS errors.
- Simple, friendly, non-technical English. No forbidden wording (see the shared safety constants; [14-ai-safety.md](./14-ai-safety.md)).
- Adding a language means adding a dictionary matching `TranslationKey`; no hardcoded fallbacks.
- RTL-ready: start/end utilities, never left/right ([13-accessibility.md](./13-accessibility.md)).

---

## Part B — Backend messaging: `messageKey` discipline

Every user-facing API error is a typed `AppError` carrying a `messageKey` of the form **`errors.<feature>.<key>`** — lowerCamelCase feature segment matching a real module under `apps/api/src/modules/`, lowerCamelCase key. The global exception filter maps it into the sanitized envelope ([26-error-handling-and-exceptions.md](./26-error-handling-and-exceptions.md)); the **frontend** maps the key to localized copy from the dictionary. The backend never localizes and never returns a sentence as the contract.

### Rules

1. **Format:** `errors.<feature>.<key>` — e.g. `errors.game.consentRequired`, `errors.fileSecurity.invalidMagicBytes`, `errors.ai.providerFailed`.
2. **One key per scenario.** Validation, consent, size, type, safety-rejection, provider-failure each get a **distinct** key — never a generic `errors.<feature>.error`. Distinct keys are what let the UI say something actionable.
3. **Keys are constants, never inline literals** (rule 12 of rules/00) — they live in `model/<feature>.constants.ts` or `packages/shared/src/constants`, and the frontend dictionary carries a translation for every key the API can emit.
4. **The developer-facing `message` is for logs only.** The client contract is `statusCode` + `errorCode` (legacy) + `messageKey`; the UI prefers `messageKey` and falls back to `errorCode` mapping for older clients.
5. **A new key ships with its dictionary entry in the same change.** A `messageKey` the frontend can't translate is an incomplete change — the parity is asserted in tests.

```ts
// model/game.constants.ts — keys are named constants, owned by the feature
export const GAME_MESSAGE_KEYS = {
  CONSENT_REQUIRED: 'errors.game.consentRequired',
  ANALYSIS_TIMED_OUT: 'errors.game.analysisTimedOut',
} as const;

// Do — throw with a scenario-specific key constant
throw new ValidationError('Consent flag missing', GAME_MESSAGE_KEYS.CONSENT_REQUIRED);

// Don't
throw new Error('Consent required');            // ✗ no key, sentence as contract
throw new ValidationError('…', 'errors.game.error'); // ✗ generic catch-all
```

### Governance

- Keys are **append-mostly**: renaming/removing one is a contract change — update the dictionary and note it in release notes.
- No orphan keys: a key with no thrower, and a thrower with no dictionary entry, are both defects ([23-review-checklist.md](./23-review-checklist.md)).

---

## Checklist

- [ ] No hardcoded user-facing string anywhere — dictionary only, `t(key)` rendering
- [ ] Every new throw path carries a distinct `errors.<feature>.<key>` constant
- [ ] The `<feature>` segment matches a real module; keys live in constants files
- [ ] Every emittable `messageKey` has a dictionary entry in the same change
- [ ] No forbidden wording in any copy; RTL-safe utilities used
